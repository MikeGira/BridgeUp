BridgeUp — Human Needs Operating System
Master Build Prompt v6.0 — Complete Global Edition with Voice
Project Name: BridgeUp
Tagline: Post your need. Get matched. Get help. No hassle.
Vision: Build a full-stack, mobile-first Progressive Web App (PWA) called BridgeUp — an AI-powered platform that matches people in need with verified helpers, resources, jobs, and training in real time. Think Uber and Google Maps combined but for solving poverty, unemployment, food insecurity, and resource gaps. Must work in any language, on smartphones AND feature phones without internet, AND through voice for users with zero literacy. Designed for underserved communities in the USA, Canada, Rwanda, across Africa, and globally.

🎙️ VOICE INTERFACE — ZERO LITERACY SUPPORT
This is a core equal feature. Every interaction available through text must also be available through voice. A zero-literacy farmer in Rwanda must access the same help as a tech-savvy Toronto user — through speaking and listening instead of reading and typing.
Browser Voice — Smartphone PWA

Speech to text: Web Speech API — SpeechRecognition — built into all browsers, free, no API key, no account
AI processing: Claude API — already in stack, understands Kinyarwanda, Swahili, French, English, Arabic, Spanish and 90+ languages natively
Text to speech: Web Speech Synthesis API — SpeechSynthesis — built into all browsers, free, no API key, no account
One large circular microphone button — minimum 120px diameter — centered on screen — no text needed
Button states with visual and audio feedback:

Idle: solid green #16A34A with subtle pulse — ready to listen
Listening: amber #F59E0B with radiating ripple rings — hearing the user
Processing: spinning indicator — thinking
Speaking: animated sound wave bars — talking back


Short chime sound plays when listening starts — confirms to user the app heard the tap
Live transcription shown as faint text below button — helps literate users verify
After 2 seconds of silence, recognition ends automatically
Claude detects language from speech and responds in the SAME language always
Response spoken aloud via SpeechSynthesis — user never needs to read anything
After speaking, button returns to idle — ready for next input
All need categories shown as large icons not text: food=bowl icon, jobs=briefcase, housing=house, medical=cross, training=book, funding=hand with coin
Every button reads itself aloud when tapped — full screen reader behavior
If internet lost during voice: plays pre-cached audio "No internet. Your message saved and will send when connection returns"
Claude system prompt for voice mode: "You are BridgeUp's voice assistant. The user cannot read or write. Speak in simple, warm, conversational {detected_language}. Never use complex words. Keep each response under 3 sentences. Ask only one question at a time. Confirm back what you understood before proceeding."

Feature Phone Voice — Twilio IVR powered by Claude

Set up Twilio Voice phone numbers — one per country: Rwanda, Canada/USA, Kenya
When user calls, Twilio greets in caller's language based on country code:

Rwanda +250: Kinyarwanda — "Murakaza neza kuri BridgeUp. Vuga ikibazo cyawe."
Kenya +254: Swahili — "Karibu BridgeUp. Sema unahitaji nini."
Canada/USA +1: English — "Welcome to BridgeUp. Tell me what you need."


Twilio records speech with transcription enabled
Transcription sent to Claude API via webhook at /voice/process
Claude responds in maximum 2 simple sentences in caller's language
Twilio speaks response back using <Say> verb with appropriate language voice
Conversation continues maximum 5 turns
After matching: either connects caller directly to helper via Twilio <Dial> or sends SMS with helper contact
No menu buttons ever — entire call is natural speech
Voice webhook routes:

POST /voice/answer — handles incoming call, plays greeting, starts recording
POST /voice/process — receives transcription, sends to Claude, returns TwiML spoken response
POST /voice/complete — ends conversation, triggers matching, sends SMS confirmation
